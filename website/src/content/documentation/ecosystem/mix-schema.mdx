# mix_schema

`mix_schema` connects AI agent protocols to your Mix design system so that AI-generated UI renders with the same brand quality as hand-crafted UI.

## The problem: every AI UI looks the same

An AI agent sends structured payloads describing *what* to render — a card, a heading, a button. Without a design system, those payloads produce generic Material widgets. Every app using the same protocol renders identical output. No brand. No differentiation.

```dart
// What the agent sends
{ "type": "Card", "children": [
    { "type": "Text", "text": "Meeting Notes", "usageHint": "h2" },
    { "type": "Button", "label": "Review", "variant": "primary" }
]}

// What renders: default Material. No brand. No design system.
Card(
  child: Column(children: [
    Text('Meeting Notes', style: Theme.of(context).textTheme.headlineSmall),
    FilledButton(onPressed: () {}, child: Text('Review')),
  ]),
)
```

The agent tells you **what** to show. Nobody tells it **how** things should look.

## The gap: two worlds, no bridge

Flutter has two mature stacks that operate independently:

```
 AGENT PROTOCOLS                    DESIGN SYSTEMS
 ──────────────────                 ──────────────────
 A2UI  /  AG-UI                     Mix  /  Material Tokens
 GenUI SDK                          ThemeExtensions

 + Component structure              + Design tokens
 + Data binding                     + Responsive variants
 + Agent communication              + Animation pipeline
 + Event routing                    + Brand consistency

 - No tokens                        - No schema input
 - No styling                       - No agent awareness
 - No responsive                    - No wire adapters
 - No animation                     - No trust gating
```

Agent protocols know *what* to render but have zero styling capability. Design systems know *how* things look but have no way to accept structured input from an agent. The gap between these two stacks is where `mix_schema` operates.

## One pipeline

Mix already provides a complete resolution pipeline: `Styler → Spec → Widget`. Developer-authored code enters this pipeline through stylers. `mix_schema` feeds AI payloads into the **same pipeline** through adapters.

```
 Developer Code ──────────────> Styler ──> Spec ──> Widget ──> Branded UI
                                  ^
 Agent Payload ──> Adapter ──> Styler +    (same from here on)
```

Both paths resolve through the same token system, the same variants, and the same widget builders. The output is indistinguishable.

```dart
// What a developer writes today
final card = BoxStyler()
    .color($surface())
    .borderRounded($radius())
    .paddingAll($spacing())
    .onDark(BoxStyler.color($surfaceDark()));

Box(style: card, child: content);
```

```dart
// What mix_schema generates from the same agent payload
Widget handleBox(SchemaNode node, RenderContext ctx) {
  final style = BoxStyler()
      .color(resolveToken(node.props['color'], ctx))
      .borderRounded(resolveToken(node.props['radius'], ctx))
      .paddingAll(resolveToken(node.props['padding'], ctx));

  return Box(style: style, child: renderChildren(node, ctx));
}
// Same BoxStyler. Same token resolution. Same pipeline.
```

## Brand safety through tokens

<Callout type="info">
  Agents send token **references**, not raw values. Your `MixScope` resolves those references to your brand's actual colors, typography, and spacing. The agent never has direct access to rendering values.
</Callout>

```dart
// 1. Define your brand tokens
final $primary = ColorToken('brand.primary');
final $surface = ColorToken('brand.surface');
final $heading = TextStyleToken('brand.heading');

// 2. Provide values at app root
MixScope(
  colors: {
    $primary: Color(0xFF6750A4),
    $surface: Color(0xFFF7F2FA),
  },
  textStyles: {
    $heading: TextStyle(fontFamily: 'Acme', fontSize: 24),
  },
  child: MyApp(),
)

// 3. Agent sends token references, not raw values
{ "color": { "type": "color", "name": "brand.primary" } }

// 4. mix_schema resolves: $primary() → your brand color
//    Unknown tokens → validation diagnostic, not a crash
```

The brand stays consistent because the agent can only *reference* tokens your design system defines. It cannot inject arbitrary colors, fonts, or sizes.

## What this unlocks

Because AI-generated UI resolves through the full Mix pipeline, it automatically inherits every capability Mix provides — without the agent knowing about any of it:

```dart
final cardStyle = BoxStyler()
    .color($surface())
    .borderRounded($radius())
    .paddingAll($spacing())

    // Responsive — breakpoint variants
    .onMobile(BoxStyler.paddingAll(8).borderRounded(8))
    .onDesktop(BoxStyler.paddingAll(24).borderRounded(16))

    // Dark mode — context variants
    .onDark(BoxStyler.color($surfaceDark()))

    // Interaction states
    .onPressed(BoxStyler.color($primaryPressed()).scale(0.98))
    .onHovered(BoxStyler.color($primaryHovered()))

    // Implicit animation
    .animate(duration: Duration(milliseconds: 200), curve: Curves.easeOut);
```

The developer controls all of this through tokens and variants. The agent controls none of it. The user sees branded, responsive, animated UI that looks identical to hand-crafted UI.

## Protocol-agnostic by design

`mix_schema` uses a wire adapter layer to convert protocol payloads into a canonical AST (`UiSchemaRoot`). When protocols change — A2UI today, AG-UI tomorrow — only the adapter changes. The rest of the pipeline stays the same.

The validation and trust layers sit between the adapter and the renderer:

- **Schema validator** checks structural correctness of the AST
- **Trust policy gate** evaluates action risk before rendering:
  - Low-risk actions execute directly
  - Medium-risk actions require confirmation
  - High-risk actions are blocked

<Callout type="warning">
  `mix_schema` treats trust policies as runtime contracts, not optional checks. Invalid or blocked payloads produce structured diagnostics instead of crashes.
</Callout>
